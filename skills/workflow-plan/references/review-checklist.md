# 审查清单详情

## REVIEW-1 审查清单（分析审查）

### 需求覆盖度

- [ ] 所有 FR（功能需求）都已在 analysis.md 中列出
- [ ] 所有 NFR（非功能需求）都已在 analysis.md 中列出
- [ ] 技术约束已完整识别
- [ ] FR/NFR 包含来源、验收方式、优先级
- [ ] 覆盖度 ≥ 95%

### 调研完整性

- [ ] 所有标记"需调研"的主题都已在 research.md 中完成
- [ ] 每个调研主题都有明确结论
- [ ] 调研结论有可靠来源支撑（至少 1 个来源）
- [ ] 技术方案对比完整（至少 2 个方案）
- [ ] 已定义评价准则，并建立权衡决策矩阵

### 证据质量

- [ ] 每个调研主题标注证据等级（A/B/C/D）与置信度
- [ ] 关键决策基于高可信来源（A/B）

### 决策点状态

- [ ] 所有待决策点都有结论或明确标记为"待讨论"
- [ ] 没有遗漏的技术决策点

### 判定规则

| 判定 | 条件 |
|------|------|
| **PASS** | 所有检查项通过，覆盖度 ≥ 95% |
| **NEEDS_ANALYZE** | 需求分析不完整（缺失 FR/NFR/约束） |
| **NEEDS_RESEARCH** | 调研不充分（缺失结论、来源不足、方案对比不完整） |

## REVIEW-2 审查清单（设计审查）

### 架构完整性

- [ ] 架构图包含所有主要模块
- [ ] 模块间关系清晰
- [ ] 每个模块有明确的职责描述（不超过 50 字）
- [ ] 架构覆盖所有功能需求
- [ ] 架构视角说明完整（上下文/容器/组件/部署）

### 技术选型

- [ ] 每个技术选型都有理由说明（至少 1 个技术论据）
- [ ] 技术选型与调研结论一致
- [ ] 提供备选方案
- [ ] 无与项目约束冲突的选型

### 依赖分析

- [ ] 内部依赖关系完整
- [ ] 外部依赖列表完整（包含版本）
- [ ] 依赖维护状态已评估
- [ ] 无已知安全问题的依赖

### 风险评估

- [ ] 识别 3-5 个关键风险（简单功能可为 1-2 个）
- [ ] 每个风险有可能性和影响评估
- [ ] 每个风险有缓解策略

### 安全与运维

- [ ] 安全与合规考虑明确（认证/授权/数据保护/审计）
- [ ] 可观测性与运维方案完整（指标/日志/追踪/告警）
- [ ] 上线/迁移/回滚策略明确

### ADR 完整性

- [ ] 关键架构决策都有 ADR 记录
- [ ] ADR 包含：状态、上下文、决策、后果
- [ ] ADR 关联到相关 FR/NFR

### 可追溯性

- [ ] 模块说明表包含"关联需求"列
- [ ] ADR 包含"关联需求"字段
- [ ] 所有 FR 都被设计覆盖
- [ ] 覆盖度 ≥ 95%

### 其他检查

- [ ] 无可扩展性问题
- [ ] 无明显安全风险
- [ ] 模块职责无重叠

### 判定规则

| 判定 | 条件 |
|------|------|
| **PASS** | 所有检查项通过，覆盖度 ≥ 95% |
| **NEEDS_RESEARCH** | 技术选型需要更多调研支撑 |
| **NEEDS_PLAN** | 设计有问题需修改（架构不完整、风险评估不足等） |

## 问题严重程度定义

| 级别 | 定义 | 影响 |
|------|------|------|
| **BLOCKER** | 阻止继续（严重安全问题、架构根本性错误） | 必须立即修复 |
| **CRITICAL** | 影响核心功能（关键模块缺失、重要风险未识别） | >2个则 REJECTED |
| **MAJOR** | 影响可维护性（文档不完整、追溯性缺失） | >5个则 NEEDS_IMPROVEMENT |
| **MINOR** | 可选优化（格式问题、表述不清晰） | 不影响判定 |
